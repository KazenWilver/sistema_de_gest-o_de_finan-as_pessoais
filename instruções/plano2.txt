## 1. ANÁLISE DO LAB_4

### 1.1 Enquadramento
O Lab #04 exige a construção de sistemas web com qualidade profissional, priorizando
organização estrutural, separação de responsabilidades, manutenibilidade e evolução
progressiva. O uso de IA é permitido **como instrumento de apoio ao desenvolvimento**,
não como substituto do raciocínio técnico. O aluno deve compreender, validar e adaptar
tudo o que é gerado.

### 1.2 Requisitos Obrigatórios (extraídos diretamente do lab_4)

| Requisito | Tipo | Aplicação ao SGFP |
|---|---|---|
| Autenticação completa (registo, login, logout, recuperação de senha) | Obrigatório | ✅ Implementar |
| Base de dados relacional ≥ 3 tabelas com CRUD | Obrigatório | ✅ 6 tabelas planeadas |
| Integração com ≥ 1 API externa (quando aplicável) | Obrigatório | ✅ API de câmbio |
| Interface responsiva | Obrigatório | ✅ Angular + CSS responsivo |
| Modo claro e modo escuro com alternância funcional | Obrigatório | ✅ ThemeService |
| Suporte a múltiplos idiomas (mínimo 2) | Obrigatório | ✅ PT + EN via ngx-translate |
| Exportação de dados ou relatórios (CSV ou PDF) | Obrigatório | ✅ CSV + PDF |
| Distinção de tipos de utilizador + permissões diferenciadas | Condicional | ✅ Justificado (admin/user) |

### 1.3 Stack Tecnológica Obrigatória
- **Frontend**: Angular
- **Backend**: PHP puro (sem frameworks)
- **Base de dados**: SQL relacional (MySQL)
- **Comunicação**: HTTP com respostas JSON

1. Funcionamento dos sistemas

3. Organização do código (principal critério)
4. Uso adequado de base de dados e APIs

## 2. DECISÕES TÉCNICAS E RESPOSTAS ÀS QUESTÕES

### 2.1 Web Scraping para Criptomoedas
**[NÃO APLICÁVEL]** — O lab_4 não menciona web scraping em nenhum ponto.
O requisito de integração externa refere-se a **API externa**. O "Sistema de
Monitoramento de Criptomoedas" é um sistema **separado** do primeiro conjunto.
O SGFP não tem relação direta com criptomoedas. Usar web scraping introduziria
dependência frágil, risco legal e complexidade desnecessária não sustentada pelo lab.
→ **Decisão**: usar API de taxas de câmbio para conversão de moedas, que é
diretamente relevante para gestão financeira pessoal.

### 2.2 Uso de Inteligência Artificial no Sistema
O lab_4 usa IA como **ferramenta de desenvolvimento** (assistente de código),
não exige funcionalidades de IA no produto final. Contudo, **não proíbe** features de IA.

**[PRESSUPOSTO EXPLÍCITO]**: Será adicionada uma funcionalidade de IA opcional
(análise de padrões financeiros) utilizando a **API do Claude (Anthropic)** no backend
PHP. Esta funcionalidade enriquece o sistema sem violar nenhuma diretriz do lab.
- O backend envia o resumo financeiro do utilizador para a API Claude
- A resposta retorna insights/recomendações em linguagem natural
- Feature isolada num `AIService.php` — removível sem impacto no resto do sistema

### 2.3 APIs Externas Selecionadas

| API | Uso | Plano | Chave? |
|---|---|---|---|
| **Frankfurter.app** | Taxas de câmbio em tempo real | 100% gratuito, open-source | Não |
| **ngx-translate** | i18n Angular (biblioteca, não API) | Open-source | N/A |
| **Claude API (Anthropic)** | Insights financeiros via IA | Free tier disponível | Sim |

**Justificação do Frankfurter.app**: sem registo, sem chave, sem limites documentados,
mantido pelo BCE (European Central Bank), retorna JSON limpo. Ideal para ambiente
académico. URL: `https://api.frankfurter.app/latest?from=USD&to=AOA,EUR`

**Justificação da Claude API**: plan gratuito com créditos iniciais suficientes para
um projeto académico. Alternativa reversível: desativar o `AIService` e o feature
permanece latente sem quebrar o sistema.

### 2.4 Exportação de Dados
- **CSV**: gerado no backend PHP nativamente (sem biblioteca)
- **PDF**: gerado no backend com **FPDF** (biblioteca PHP gratuita, zero dependências externas)
- Ambos os formatos expostos via endpoints dedicados com cabeçalhos HTTP corretos

### 3.2 Definição das Tabelas (SQL — MySQL)

```sql
-- ============================================================
-- TABELA: users
-- ============================================================
CREATE TABLE users (
    id            INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    name          VARCHAR(100)  NOT NULL,
    email         VARCHAR(150)  NOT NULL UNIQUE,
    password      VARCHAR(255)  NOT NULL,               -- bcrypt
    role          ENUM('admin','user') NOT NULL DEFAULT 'user',
    currency      VARCHAR(10)   NOT NULL DEFAULT 'AOA',
    language      VARCHAR(5)    NOT NULL DEFAULT 'pt',
    reset_token   VARCHAR(255)  NULL,
    token_expires DATETIME      NULL,
    created_at    TIMESTAMP     DEFAULT CURRENT_TIMESTAMP,
    updated_at    TIMESTAMP     DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- ============================================================
-- TABELA: categories
-- ============================================================
CREATE TABLE categories (
    id         INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    user_id    INT UNSIGNED NOT NULL,
    name       VARCHAR(100) NOT NULL,
    type       ENUM('income','expense') NOT NULL,
    icon       VARCHAR(60)  NOT NULL DEFAULT 'circle',
    color      VARCHAR(7)   NOT NULL DEFAULT '#6366f1',
    is_default TINYINT(1)   NOT NULL DEFAULT 0,
    created_at TIMESTAMP    DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT fk_cat_user FOREIGN KEY (user_id)
        REFERENCES users(id) ON DELETE CASCADE
);

-- ============================================================
-- TABELA: transactions
-- ============================================================
CREATE TABLE transactions (
    id          INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    user_id     INT UNSIGNED   NOT NULL,
    category_id INT UNSIGNED   NOT NULL,
    type        ENUM('income','expense') NOT NULL,
    amount      DECIMAL(15,2)  NOT NULL,
    currency    VARCHAR(10)    NOT NULL DEFAULT 'AOA',
    amount_base DECIMAL(15,2)  NOT NULL,               -- valor na moeda base do utilizador
    description VARCHAR(255)   NOT NULL,
    date        DATE           NOT NULL,
    notes       TEXT           NULL,
    created_at  TIMESTAMP      DEFAULT CURRENT_TIMESTAMP,
    updated_at  TIMESTAMP      DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    CONSTRAINT fk_tx_user FOREIGN KEY (user_id)
        REFERENCES users(id) ON DELETE CASCADE,
    CONSTRAINT fk_tx_cat FOREIGN KEY (category_id)
        REFERENCES categories(id) ON DELETE RESTRICT
);

-- ============================================================
-- TABELA: budgets
-- ============================================================
CREATE TABLE budgets (
    id          INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    user_id     INT UNSIGNED   NOT NULL,
    category_id INT UNSIGNED   NULL,                   -- NULL = orçamento global
    name        VARCHAR(100)   NOT NULL,
    amount      DECIMAL(15,2)  NOT NULL,
    period      ENUM('weekly','monthly','yearly') NOT NULL DEFAULT 'monthly',
    start_date  DATE           NOT NULL,
    end_date    DATE           NOT NULL,
    created_at  TIMESTAMP      DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT fk_bud_user FOREIGN KEY (user_id)
        REFERENCES users(id) ON DELETE CASCADE,
    CONSTRAINT fk_bud_cat FOREIGN KEY (category_id)
        REFERENCES categories(id) ON DELETE SET NULL
);

-- ============================================================
-- TABELA: goals
-- ============================================================
CREATE TABLE goals (
    id             INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    user_id        INT UNSIGNED   NOT NULL,
    name           VARCHAR(100)   NOT NULL,
    target_amount  DECIMAL(15,2)  NOT NULL,
    current_amount DECIMAL(15,2)  NOT NULL DEFAULT 0.00,
    deadline       DATE           NULL,
    status         ENUM('active','completed','cancelled') NOT NULL DEFAULT 'active',
    created_at     TIMESTAMP      DEFAULT CURRENT_TIMESTAMP,
    updated_at     TIMESTAMP      DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    CONSTRAINT fk_goal_user FOREIGN KEY (user_id)
        REFERENCES users(id) ON DELETE CASCADE
);

-- ============================================================
-- TABELA: currency_rates (cache de taxas de câmbio)
-- ============================================================
CREATE TABLE currency_rates (
    id              INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    base_currency   VARCHAR(10)    NOT NULL,
    target_currency VARCHAR(10)    NOT NULL,
    rate            DECIMAL(20,6)  NOT NULL,
    fetched_at      TIMESTAMP      DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uq_pair (base_currency, target_currency)
);
```

### 3.3 Normalização
- **1NF**: todos os campos são atómicos, sem grupos repetitivos
- **2NF**: todas as colunas dependem da chave primária completa
- **3NF**: sem dependências transitivas (ex.: `currency` em `transactions` é dado
  da transação, não derivado de outra tabela não-chave)
- `currency_rates` serve apenas de cache com TTL gerido no `CurrencyService.php`

### 4.2 Router Central (public/index.php)

```php
<?php
// public/index.php — Router central

require_once __DIR__ . '/../src/Helpers/Response.php';
require_once __DIR__ . '/../src/Middleware/CorsMiddleware.php';

CorsMiddleware::handle();

$method = $_SERVER['REQUEST_METHOD'];
$uri    = parse_url($_SERVER['REQUEST_URI'], PHP_URL_PATH);
$uri    = rtrim($uri, '/');
$parts  = explode('/', ltrim($uri, '/'));
// Ex: ['api', 'transactions', '5']

// Mapa de rotas: [METHOD][pattern] => [Controller, action]
// Cada controller é carregado apenas quando necessário (lazy load manual)

$routes = [
    'POST'   => [
        'api/auth/register'          => ['AuthController',        'register'],
        'api/auth/login'             => ['AuthController',        'login'],
        'api/auth/logout'            => ['AuthController',        'logout'],
        'api/auth/forgot-password'   => ['AuthController',        'forgotPassword'],
        'api/auth/reset-password'    => ['AuthController',        'resetPassword'],
        'api/transactions'           => ['TransactionController', 'store'],
        'api/categories'             => ['CategoryController',    'store'],
        'api/budgets'                => ['BudgetController',      'store'],
        'api/goals'                  => ['GoalController',        'store'],
        'api/goals/:id/contribute'   => ['GoalController',        'contribute'],
        'api/ai/insights'            => ['AIController',          'getInsights'],
    ],
    'GET'    => [
        'api/auth/me'                => ['AuthController',        'me'],
        'api/transactions'           => ['TransactionController', 'index'],
        'api/transactions/:id'       => ['TransactionController', 'show'],
        'api/categories'             => ['CategoryController',    'index'],
        'api/budgets'                => ['BudgetController',      'index'],
        'api/budgets/:id/progress'   => ['BudgetController',      'progress'],
```

### 4.3 Separação de Responsabilidades — Exemplo Concreto

```
Request POST /api/transactions
      │
      ▼
AuthMiddleware::handle()          ← valida JWT, extrai user_id
      │
      ▼
TransactionController::store()    ← recebe JSON, valida formato básico
      │
      ▼
TransactionService::create()      ← aplica regras de negócio:
                                     · confirmar que category pertence ao user
                                     · converter amount para moeda base via CurrencyService
                                     · verificar se excede budget ativo
      │
      ▼
TransactionRepository::insert()   ← executa INSERT com PDO prepared statement
      │
      ▼
Response::json(['id' => $id], 201)
```

```typescript
// features/transactions/services/transaction.service.ts

@Injectable({ providedIn: 'root' })
export class TransactionService {
  private readonly endpoint = `${environment.apiUrl}/transactions`;
  private transactions$ = new BehaviorSubject<Transaction[]>([]);

  constructor(private http: HttpClient) {}

  getAll(filters?: TransactionFilter): Observable<Transaction[]> {
    const params = this.buildParams(filters);
    return this.http.get<ApiResponse<Transaction[]>>(this.endpoint, { params })
      .pipe(
        map(res => res.data),
        tap(data => this.transactions$.next(data))
      );
  }

  create(payload: CreateTransactionDto): Observable<Transaction> {
    return this.http.post<ApiResponse<Transaction>>(this.endpoint, payload)
      .pipe(map(res => res.data));
  }

  update(id: number, payload: Partial<CreateTransactionDto>): Observable<Transaction> {
    return this.http.put<ApiResponse<Transaction>>(`${this.endpoint}/${id}`, payload)
      .pipe(map(res => res.data));
  }

  remove(id: number): Observable<void> {
    return this.http.delete<void>(`${this.endpoint}/${id}`);
  }

  private buildParams(filters?: TransactionFilter): HttpParams { /* ... */ }
}
```

### 5.4 Modo Claro/Escuro

```typescript
// core/services/theme.service.ts
@Injectable({ providedIn: 'root' })
export class ThemeService {
  private theme$ = new BehaviorSubject<'light' | 'dark'>('light');

  constructor(private storage: StorageService) {
    const saved = this.storage.get('theme') ?? 'light';
    this.apply(saved as 'light' | 'dark');
  }

  toggle(): void {
    const next = this.theme$.value === 'light' ? 'dark' : 'light';
    this.apply(next);
  }

  private apply(theme: 'light' | 'dark'): void {
    document.documentElement.setAttribute('data-theme', theme);
    this.storage.set('theme', theme);
    this.theme$.next(theme);
  }
}
```

## 7. FUNCIONALIDADES DO SISTEMA

### 7.1 Autenticação
- Registo com validação de email único
- Login com JWT (expiração configurável, ex: 7 dias)
- Logout (token invalidado no frontend; opcional: blacklist no backend)
- Recuperação de senha via link com token temporário (armazenado em BD)

### 7.2 Dashboard
- Saldo atual (soma receitas − despesas do mês corrente)
- Cards: Total Receitas / Total Despesas / Poupança do mês
- Lista das últimas 5 transações
- Progress bars dos orçamentos ativos
- Widget de insight IA (resumo textual da saúde financeira)

### 7.3 Transações
- CRUD completo com validação no frontend e backend
- Filtros: período, tipo (receita/despesa), categoria
- Paginação server-side
- Ordenação por data, valor, categoria

### 7.4 Orçamentos
- Definição de limite por categoria ou global
- Período: semanal / mensal / anual
- Indicador visual de progresso (cor muda: verde → amarelo → vermelho)
- Alerta quando > 80% consumido

### 7.5 Metas de Poupança
- Criar meta com valor-alvo e prazo
- Contribuições manuais ao longo do tempo
- Progress bar com % atingida e projeção de data de conclusão

### 7.6 Categorias
- CRUD de categorias personalizadas por utilizador
- Categorias padrão criadas automaticamente no registo
- Ícone e cor personalizáveis

### 7.7 Relatórios
- Gráfico de linha: receitas vs despesas nos últimos N meses
- Gráfico de pizza: distribuição por categoria
- Tabela de resumo mensal
- Exportação filtrada (data de início / fim) em CSV ou PDF

### 7.8 Tipos de Utilizador (Condicional — justificado)
- **user**: acesso ao próprio espaço financeiro
- **admin**: acesso ao painel de administração (lista de utilizadores, estatísticas globais do sistema)
- O role=admin é atribuído diretamente na BD (sem registo público como admin)
