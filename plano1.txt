### 1. Análise de Requisitos e Essência do Projeto

A essência de um "Sistema de Gestão Financeira Pessoal" é permitir ao utilizador registar, categorizar e analisar as suas receitas e despesas.

**Requisitos Obrigatórios Extraídos (Aplicados ao Contexto):**

* **Stack:** Angular (Frontend), PHP puro (Backend), SQL relacional.
* **Comunicação:** REST HTTP com JSON.
* **Autenticação:** Registo, Login, Logout, Recuperação de senha (baseada em token guardado na BD).
* **Base de Dados:** Mínimo de 3 tabelas e suporte a operações CRUD (foco nas transações financeiras).
* **Integrações:** Pelo menos uma API Externa.
* **Interface:** Responsiva, Light/Dark mode, Multi-idioma (mínimo PT e EN).
* **Exportação:** Relatório financeiro em CSV ou PDF.
* **Gestão de Acessos:** Admin vs. Utilizador Comum.

---

### 2. Estrutura de Base de Dados (SQL Relacional)

Para cumprir o mínimo de três tabelas, garantir normalização (3FN) e suportar as funcionalidades financeiras e de permissões:

1. **Tabela `users**` (Autenticação e Perfis)
* `id` (PK, INT, Auto Increment)
* `name` (VARCHAR)
* `email` (VARCHAR, UNIQUE)
* `password_hash` (VARCHAR)
* `role` (ENUM: 'admin', 'user') -> *Cumpre a exigência de distinção de utilizadores.*
* `reset_token` (VARCHAR, NULL)
* `created_at` (TIMESTAMP)


2. **Tabela `categories**` (Classificação de Transações)
* `id` (PK, INT, Auto Increment)
* `name` (VARCHAR, ex: "Alimentação", "Salário")
* `type` (ENUM: 'income', 'expense')
* `user_id` (FK, INT, NULL) -> *Se NULL, é uma categoria global do sistema. Se preenchido, é uma categoria personalizada criada pelo utilizador.*


3. **Tabela `transactions**` (Operações CRUD Principais)
* `id` (PK, INT, Auto Increment)
* `user_id` (FK, INT) -> *Refere `users.id`.*
* `category_id` (FK, INT) -> *Refere `categories.id`.*
* `amount` (DECIMAL(10,2))
* `description` (VARCHAR)
* `transaction_date` (DATE)
* `created_at` (TIMESTAMP)

### 3. Arquitetura do Backend (PHP Puro)

Manteremos o padrão *Front Controller* para garantir um único ponto de entrada, separando rotas, controladores, modelos e serviços.

**Organização de Diretórios (Backend):**

```text
/backend
 ├── /config
 │    └── database.php       # Conexão PDO (Singleton)
 ├── /controllers            # Lógica de interface (Endpoints)
 │    ├── AuthController.php
 │    ├── TransactionController.php
 │    ├── CategoryController.php
 │    └── ExportController.php
 ├── /models                 # Acesso a Dados
 │    ├── User.php
 │    ├── Category.php
 │    └── Transaction.php
 ├── /services               # Lógica pesada e Integrações
 │    ├── CurrencyApiService.php # Chamadas à API de câmbio
 │    └── ReportService.php      # Geração de CSV/PDF
 ├── /core                   # Utilitários
 │    ├── Router.php         # Roteamento
 │    └── JwtAuth.php        # Validação de Tokens JWT
 ├── .htaccess               # Redirecionamento para index.php
 └── index.php               # Bootstrap

```

### 4. Arquitetura do Frontend (Angular)

O Frontend focará em dashboards visuais (para análise de gastos) e formulários de introdução de dados.

**Organização de Diretórios (Frontend):**

```text
/frontend/src/app
 ├── /core                   # Lógica Global e Segurança
 │    ├── /guards            # AuthGuard, AdminGuard
 │    ├── /interceptors      # JwtInterceptor
 │    └── /services          # AuthService, ThemeService, I18nService
 ├── /shared                 # Componentes de Interface Pura
 │    ├── /components        # navbar, sidebar, modal, chart-widget
 │    └── /pipes             # currency-formatter, translate
 ├── /features               # Lógica de Negócio UI (Módulos)
 │    ├── /auth              # login, register
 │    ├── /dashboard         # Resumo financeiro (gráficos de despesas/receitas)
 │    └── /transactions      # Tabela CRUD com filtros e adição de registos
 ├── app-routing.module.ts
 └── app.module.ts

```

**Navegação e Permissões:**

* **User:** Dashboard (Resumo do mês), Transações (Lista, Adicionar, Editar, Eliminar), Exportar Relatório (PDF do mês).
* **Admin:** Dashboard global do sistema (número de utilizadores ativos), Gestão de Categorias Globais.

### 5. Decisões Estratégicas: APIs, Scraping e IA

**Qual API Externa Usar?**

* **Abordagem Recomendada: ExchangeRate-API (ou Frankfurter API).**
* *Justificação:* O documento exige uma API externa "quando aplicável". Num sistema financeiro, uma funcionalidade extremamente útil é a **Conversão de Moedas** (ex: o utilizador registou despesas em Kwanzas (AOA) e quer ver o equivalente em Dólares (USD) ou Euros (EUR) no seu dashboard). Estas APIs são gratuitas, não exigem autenticação complexa (Frankfurter é open-source e não requer chave) e respondem em JSON perfeito para o nosso `CurrencyApiService`.

**Uso de Web Scraping:**

* **Decisão: NÃO UTILIZAR.**
* *Justificação:* Não há necessidade técnica de *scraping* neste projeto. As taxas de câmbio são melhor obtidas via REST API, garantindo robustez e conformidade com as boas práticas de Engenharia de Software.

**Uso de Inteligência Artificial no Sistema:**

* **Durante o Desenvolvimento:** Conforme o guião, uso obrigatório e crítico para auxiliar na estruturação e depuração do código.
* **No Sistema Final (Funcionalidade Opcional de "Luxo"):** Se desejar adicionar valor extra para a avaliação, pode implementar um "Categorizador Inteligente". No frontend, quando o utilizador escreve "Comprei um hambúrguer no KFC" na descrição, o sistema envia a frase para a API da OpenAI/Gemini (via backend PHP), que devolve a categoria id ideal (ex: "Alimentação"). *Atenção: Apenas implemente isto se as funcionalidades base (CRUD, Autenticação, Exportação) estiverem 100% concluídas e isentas de bugs.*
